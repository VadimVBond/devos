# DECISIONS.md

*   **2026-06-18: Переход к Orchestration Layer (Kernel + Plugins)**
    *   **Решение**: Отказ от монолитных модулей в пользу тонкого детерминированного ядра и системы плагинов.
    *   **Причина**: Необходимость изоляции инструментов от логики планирования и обеспечение безопасности.
    *   **Последствия**: Усложнение структуры (ядро/плагины), но высокая расширяемость.

*   **2026-06-18: Реструктуризация в корень (Flat Root Structure)**
    *   **Решение**: Перенос папок `kernel`, `cli`, `memory` из `src/devos/` в корень проекта.
    *   **Причина**: Соответствие предоставленной спецификации "Дерево.txt".
    *   **Последствия**: Упрощение импортов в CLI, плоская структура проекта.

*   **2026-06-18: Реализация плагина Git**
    *   **Решение**: Добавлен плагин `git` для управления репозиториями через subprocess.
    *   **Причина**: Расширение возможностей оркестрации для автоматизации рабочих процессов разработчика.
    *   **Последствия**: Система может выполнять полный цикл: от изменения кода до фиксации изменений в Git.

*   **2026-06-18: Реализация плагина FS и интеграция в Executor**
    *   **Решение**: Реализован первый функциональный плагин `fs` и механизм его вызова через `PluginRegistry` в `KernelExecutor`.
    *   **Причина**: Переход от симуляции к реальному выполнению задач оркестрационным слоем.
    *   **Последствия**: Система теперь способна изменять состояние окружения (файлы, папки) согласно сгенерированным планам.

*   **2026-06-18: Использование ИИ для генерации Execution Graph (DAG)**
    *   **Решение**: Планировщик использует LLM для создания графа задач с узлами и ребрами.
    *   **Причина**: Обеспечение гибкости при решении сложных многошаговых задач.
    *   **Последствия**: Необходимость строгой программной валидации графа (циклы, ID) перед исполнением.

*   **2026-06-18: Разделение AI-слоя на Execution (ai/) и Cognitive (.ai/)**
    *   **Решение**: Весь исполняемый код перенесен в `ai/`, все промпты и инструкции поведения — в `.ai/`.
    *   **Причина**: Обеспечение чистоты кода и возможности менять поведение системы без изменения программной логики.
    *   **Последствия**: Четкая граница между "как исполнять" и "как думать".

*   **2026-06-18: Использование Async SQLAlchemy + SQLite**
    *   **Решение**: Выбор `aiosqlite` и асинхронного режима SQLAlchemy.
    *   **Причина**: Поддержка asyncio-архитектуры для будущих параллельных задач.
    *   **Последствия**: Необходимость асинхронного выполнения во всех слоях взаимодействия с БД.
